# Проект Veranda: Kitchen Analytics, КухняОнлайн и Онлайн‑меню
Документ описывает продукт и требования: сначала бизнес‑логика (без технических деталей), затем полная техническая спецификация, достаточная для воспроизведения фронтенда и бэкенда “с нуля”.

---

## Часть 1. Бизнес‑ТЗ (без технических деталей)

### 1. Цели продукта
- Контролировать скорость кухни/бара в моменте и по времени суток.
- Разбирать спорные кейсы (“долго готовили”) быстро и наглядно.
- Оперативно реагировать на критические задержки без постоянного мониторинга.
- Дать поварам понятное “табло текущих блюд”.
- Управлять онлайн‑меню для гостей (видимость, переводы, публикация).

### 2. Роли и доступы
- Авторизация по Google‑аккаунтам.
- Роли:
  - Менеджер: Дашборд, Таблица, Управление.
  - Повар/бармен: КухняОнлайн (табло), без доступа к Таблице (по настройке).
  - Владелец/админ: полный доступ.
- Отдельное право “Не учитывать” — кто может исключать позиции из аналитики и табло.

### 3. Основные разделы продукта

#### 3.1. Дашборд (аналитика по часам)
- Для выбранного диапазона дат/часов показываются:
  - среднее время ожидания (по станциям: кухня/бар);
  - максимальное время ожидания;
  - распределение нагрузки по часам.
- Клик по часу открывает Таблицу с теми же фильтрами.

#### 3.2. Таблица (детализация по чекам и блюдам)
- Данные сгруппированы по чекам.
- Внутри чека видны позиции, время отправки на станцию, готовность, статус.
- Подсказки “как считается” и видимые поля, достаточные для разбора кейсов.
- Действие “Не учитывать” исключает позицию из:
  - расчётов дашборда,
  - табло КухняОнлайн,
  - Telegram‑алертов.

#### 3.3. КухняОнлайн (табло для кухни/бара)
- Экран “сейчас готовится”:
  - карточки чеков в одну линию с горизонтальным скроллом;
  - внутри карточек — только блюда, которые ещё не готовы;
  - фильтр по станции: Все / Кухня / Бар;
  - живой таймер ожидания на каждой позиции;
  - красная обводка у блюд, вышедших за допустимый лимит ожидания;
  - если активных блюд нет — “ВСЕ ЗАКАЗЫ ВЫДАНЫ”.
- Звук:
  - при появлении новой позиции на табло — короткий звуковой сигнал;
  - переключатель 🔊/🔇 в заголовке.
- “Принято” (✕) — если у пользователя есть право “Не учитывать”, позиция исключается из аналитики и убирается из табло.

#### 3.4. Telegram‑уведомления
- Цель — уведомлять о критических задержках.
- Правила:
  - лимит ожидания динамический (больше при высокой нагрузке);
  - уведомления обновляются (актуальное время ожидания) и удаляются при закрытии ситуации;
  - кнопка ✅ Принято подавляет повторы на заданный период;
  - по отдельным категориям (например, кальяны) алерты не отправляются;
  - позиции “удалено” и “не учитывать” не алертятся.

#### 3.5. Управление (админка)
- Настройки логики Telegram‑уведомлений (лимиты/нагрузка/тайм‑аут “Принято”, whitelist).
- Управление доступом пользователей (Дашборд/Таблица/КухняОнлайн/Управление/“Не учитывать”).
- Управление онлайн‑меню (категории, публикация, переводы).

## Часть 2. Техническая спецификация

### 2.1. Принципы “без легаси, сразу оптимально”
1) Единое место авторизации и прав: один middleware, единая модель permissions.
2) Единый шаблон шапки/навигации: не дублировать HTML/CSS по страницам.
3) Все обновления “живых экранов” (КухняОнлайн) — через JSON API, отдельные endpoints.
4) Минимум логики в шаблонах: расчёты и бизнес‑правила — в сервисах.
5) База: индексы на ключевые поля, запреты на “SELECT *” в горячих местах, лимиты на выборки.

### 1. Технологический стек
- Backend: PHP 8.x
- DB: MySQL 8.x
- Frontend: HTML/CSS/JS (без тяжёлых фреймворков)
- Интеграции: Poster API, Telegram Bot API, Google OAuth
- Деплой: GitHub Actions + rsync по SSH, сервер Linux (Apache/Nginx + PHP)

### 2. Конфигурация и секреты
- .env (не коммитится): DB_*, POSTER_API_TOKEN, GOOGLE_*, TELEGRAM_*, APP_URL.
- .gitignore исключает .env и логи.

### 3. Авторизация и права
- Google OAuth: в сессию сохраняются user_email, user_name, user_avatar.
- Таблица users: id, email, is_active, permissions_json, created_at.
- permissions: dashboard, rawdata, kitchen_online, admin, exclude_toggle.

### 4. Данные и расчёты

Станции: Kitchen (id 2), Bar (id 3).

Таблица kitchen_stats содержит “позиции блюд” в рамках чека.

Рекомендуемая схема (минимум):
- Идентификаторы: `id`, `transaction_date`, `transaction_id`, `receipt_number`, `table_number`, `station`, `dish_id`, `dish_name`
- Времена: `transaction_opened_at`, `transaction_closed_at`, `ticket_sent_at`, `ready_pressed_at`
- Статусы: `status`, `pay_type`, `close_reason`
- Исключения: `was_deleted`, `exclude_from_dashboard`, `exclude_auto`
- Telegram: `tg_message_id`, `tg_acknowledged`, `tg_acknowledged_at`, `tg_acknowledged_by`
- Служебное: `waiter_name`, `created_at`

Индексы (обязательно для скорости):
- `(transaction_date, status, station)`
- `(transaction_date, transaction_id)`
- `(transaction_date, ticket_sent_at)`
- `(transaction_date, exclude_from_dashboard)`
- `(transaction_date, tg_message_id)`

Правила расчёта ожидания: старт = ticket_sent_at, конец = ready_pressed_at; без конца — позиция “готовится”.

“Не учитывать” полностью исключает позицию из:
- дашборда,
- табло,
- Telegram‑алертов.

Таблица system_meta: alert_*, telegram_ack_whitelist, poster_last_sync_at.

### 5. Страницы и фронтенд‑логика

dashboard.php, kitchen_online.php, rawdata.php, admin.php, logs.php — как описано в Часть 1, с обновлением KitchenOnline каждые ~10 секунд через JSON.

### 6. Сервисные классы и интеграции
Database, PosterAPI, KitchenAnalytics, EventLogger, Auth — реализуют подключение к БД, вызовы Poster, расчёты аналитики, логирование и авторизацию.

### 7. JSON‑эндпоинты (примерные)
/kitchen_online.php?ajax=now, /rawdata.php?ajax=table, /admin.php?ajax=set_permissions, /admin.php?ajax=telegram_config, /menu endpoints (импорт/публикация/переводы).

### 8. Cron и фоновые задачи
Kitchen analytics sync (*/5), Telegram alerts (*/5), Menu sync (каждый час).

#### 2.6.2. Очистка сообщений
Скрипт в начале проходит по сообщениям с `tg_message_id` за окно **2 часа** (по `ticket_sent_at`) и удаляет устаревшие:
- ✅ Принято
- готово
- удалено
- не учитывать
- чек закрыт

#### 2.6.3. Кандидаты на отправку
Позиции:
- чек открыт (`status=1`)
- `ticket_sent_at` старше лимита ожидания
- не готово
- не удалено
- не исключено
- не кальян (категория 47)
- не заглушено “✅ Принято” (или заглушка уже истекла)

#### 2.6.4. Кнопка “✅ Принято”
Работает только для username из whitelist, ставит ack по дублям позиции в рамках чека.

### 9. Деплой и проверка
GitHub Actions: rsync по SSH, php -l на сервере; секреты SSH_HOST/PORT/USER/DEPLOY_PATH/PRIVATE_KEY.

### 10. UX‑правила и общие принципы
Минимум логики в шаблонах; единая шапка; JSON‑обновления; индексы/лимиты/запрет “SELECT *”; логи в logs.php; критичные ошибки — EventLogger.

### 2.9. Cron‑расписание (актуально)
```bash
# Analytics: синхронизация данных кухни/бара (раз в 5 минут)
*/5 * * * * /usr/bin/php /var/www/.../scripts/kitchen/sync.php >> /var/www/.../cron.log 2>&1

# Telegram: проверка и отправка алертов (раз в 5 минут)
*/5 * * * * /usr/bin/php /var/www/.../scripts/kitchen/telegram_alerts.php >> /var/www/.../telegram.log 2>&1

# Menu: обновление меню из Poster (раз в час)
0 * * * * /usr/bin/php /var/www/.../scripts/menu/sync.php >> /var/www/.../menu_sync.log 2>&1
```
