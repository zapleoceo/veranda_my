# Рефакторинг монолитных страниц в MVC + централизация API

Дата: 2026-05-05  
Область: монолитные страницы в корне репозитория (`zapara.php`, `rawdata.php`, `kitchen_online.php`, `dashboard.php`)  
Цель: разделение ответственности (MVC/SoC), устранение дублирования HTTP-эндпойнтов и Poster/SQL-запросов, при сохранении поведения UI (включая AJAX без перезагрузки).

## 1. Контекст и проблема

В проекте есть страницы, которые одновременно:
- проверяют авторизацию/права,
- обрабатывают `$_GET/$_POST` (в т.ч. `ajax`),
- выполняют SQL-запросы,
- выполняют запросы к Poster API,
- рендерят HTML,
- содержат inline-утилиты (например, собственный curl-клиент для Poster).

Это приводит к:
- высокой связанности и сложности изменений,
- дублированию API-логики (в т.ч. разный способ запросов в Poster),
- трудностям тестирования (нет границ между слоями).

## 2. Объём работ (итерация 1)

Рефакторим все найденные монолитные страницы:
- `zapara.php`
- `rawdata.php`
- `kitchen_online.php`
- `dashboard.php`

Исторические дубликаты (`admin_orig.php`, `admin_dc38ed2.php`) в этой итерации не трогаем функционально, но фиксируем как кандидаты на дальнейшую очистку.

## 3. Целевое разбиение на слои

### 3.1. View
- HTML-шаблоны выносятся в отдельные файлы внутри папки модуля.
- View не обращается напрямую ни к SQL, ни к Poster API.
- View получает только подготовленные данные для рендера.

### 3.2. Controller
- Точка входа модуля (`/<module>/index.php`) определяет режим:
  - страница (HTML)
  - `ajax` (JSON)
- Контроллер:
  - валидирует входные параметры,
  - вызывает нужный use-case/модель,
  - возвращает JSON/HTML.

### 3.3. Model/Service/Repository
- SQL-логика: `Repository`-классы, использующие `\App\Classes\Database`.
- Poster-логика: `Service`-классы, использующие `\App\Classes\PosterAPI`.
- Классы не зависят от суперглобалов и не пишут output напрямую.

## 4. Централизация API по категориям

Создаём 2 “категории” в `api/`:
- `api/sql/<module>/index.php?ajax=...` — эндпойнты, которые ходят в SQL.
- `api/poster/<module>/index.php?ajax=...` — эндпойнты, которые ходят в Poster API.

Формат роутинга сохраняем через `?ajax=` (без обязательного rewrite).

## 5. Маршрутизация и совместимость URL (гибрид)

Для каждого модуля создаём новый “чистый” URL:
- `/zapara/`
- `/rawdata/`
- `/kitchen_online/`
- `/dashboard/`

Старые URL в корне сохраняются как совместимость:
- При запросе HTML: делают 301 redirect на новый URL модуля.
- При запросах с `?ajax=...`: проксируют на новый `api/*` эндпойнт (или включают соответствующий контроллер), чтобы:
  - не ломать текущий JS,
  - можно было мигрировать фронт постепенно.

## 6. План “до/после” тестирования

### 6.1. Базовые принципы
- До удаления старых эндпойнтов фиксируем “baseline” ответы.
- После переноса проверяем новые эндпойнты тем же набором запросов.
- Сравнение “умное”, не байт-в-байт:
  - одинаковые `ok`/коды ответа,
  - наличие ключевых полей,
  - типы и размеры массивов,
  - допустимые расхождения: timestamp, порядок элементов (если UI от него не зависит).

### 6.2. Набор тестов
Для каждого модуля:
- Smoke: 2–6 запросов на основные сценарии.
- Отдельно: сценарии, которые в UI происходят без перезагрузки (AJAX/Fetch).

Тест-скрипты выполняются локально из sandbox (при наличии `POSTER_API_TOKEN` в окружении).

## 7. Итерационный порядок

1) Zapara: убрать самописный Poster curl-клиент, вынести Poster запросы в `api/poster/zapara`, отделить HTML/JS.
2) Rawdata: разделить SQL-пагинацию/тогглы (в `api/sql/rawdata`) и Poster-подтягивание справочников (в `api/poster/rawdata` при необходимости).
3) Kitchen online: вынести всё, что обновляется “живьём” (AJAX), в `api/sql/kitchen_online` + Poster в `api/poster/kitchen_online` (если используется).
4) Dashboard: вынести тяжёлые SQL-агрегации в `api/sql/dashboard` (если появятся AJAX), оставить HTML тонким.

## 8. Критерии готовности

- Для каждого модуля:
  - новый URL `/module/` работает,
  - старый `/module.php` корректно редиректит на HTML и поддерживает AJAX-совместимость,
  - все “живые” действия в UI работают как раньше,
  - Poster-запросы выполняются через единый `\App\Classes\PosterAPI` (без дублей curl),
  - SQL-запросы централизованы и не дублируются.
